# Role Involvement Protocol

Every role must introduce itself before doing task work in chat.

The goal is simple: the user should immediately know which agent is involved, what
that agent owns, and what kind of work is about to happen.

## Required Format

Put the involvement line at the top of the response:

```markdown
### <Emoji> **<Role> Involved**
```

Then continue with the task response, plan, status report, packet, or
implementation notes.

## Rules

- The involvement line comes before tool calls, shell commands, file reads,
  analysis, planning, file edits, or implementation notes.
- If a tool requires an initial assistant message before it can run, that first
  message must include the involvement line.
- If the role realizes it forgot to announce before using tools, it should stop,
  announce the correct role, and explain that the prior tool use was a protocol
  miss before continuing.
- The Office Assistant is involved automatically for any unstructured prompt. It
  does not need to be invoked by name.
- The CEO role must be involved whenever the task involves organizational, team structure, or company governance tasks, or when modifying governance files (such as `AGENTS.md`, `CEO_OVERVIEW.md`, or files under `docs/ai-office/`), or when explicitly requested by the user.
- If a user prompt requires both triage/routing and organizational governance, the main agent must involve both roles sequentially (e.g., printing both the Office Assistant and CEO involvement banners).
- If a user message begins with a specific role name and colon, that role
  is involved directly and skips the Office Assistant.
- Every native sub-agent launch or ready-to-paste packet generated by the Office
  Assistant must include the target role's involvement line as the first visible
  line in the role contract.
- If multiple roles are involved in one session, each role gets its own line.
- Status mode still stays read-only even after the Office Assistant is involved.
- Do not use the banner to claim ownership outside the role's documented scope.

## Example

```markdown

I will design the onboarding flow as a Flutter-ready contract with routes,
states, copy, responsive behavior, semantic labels, and acceptance criteria.
```
